# 远程仓库协议选型建议

> 基于已确认的远程需求，版本 v1.1，日期 2026-06-07
>
> ✅ **本方案已于 2026-06-07 被确认为项目正式技术方案。**
> 实施路线：先 SMB（电脑→手机），后 HTTP Server + mDNS（设备间）。

---

## 一、场景分析

远程图片仓库涉及三种典型场景：

| 场景 | 服务端 | 客户端 | 网络环境 |
|------|--------|--------|---------|
| 场景A | Windows/macOS/Linux 电脑 | Android 手机/平板 | 同一局域网（WiFi） |
| 场景B | Android 手机 | Android 手机 | 同一局域网（WiFi） |
| 场景C | Android 手机/平板 | Android 手机/平板 | 同一局域网（WiFi） |

---

## 二、场景A：电脑 → 手机/平板

### 推荐协议：SMB（Samba）✅ 已确认最优

| 维度 | 评价 |
|------|------|
| 成熟度 | ⭐⭐⭐⭐⭐ 极其成熟，Windows/macOS/Linux 原生支持 |
| 配置难度 | ⭐⭐ 电脑端开共享即用，手机端填入 IP/账号/密码 |
| 性能 | ⭐⭐⭐⭐ 局域网传输速度优秀，支持大文件流式读取 |
| 功能完整 | ⭐⭐⭐⭐⭐ 完整文件系统操作：目录列表、文件读取、缩略图缓存 |
| 安卓生态 | ⭐⭐⭐⭐ JCIFS / SMBJ 库成熟可用 |

**推荐技术栈：**
- 库：`org.samba.jcifs:jcifs:2.1.34` 或 `com.hierynomus:smbj:0.13.0`
- SMBJ 更推荐：纯 Java 实现，支持 SMB2/SMB3，性能优于 JCIFS

**实现要点：**
```
1. 用户在电脑上设置共享文件夹
2. 手机端输入：IP 地址、共享名、用户名、密码
3. 连接成功后，远程目录以"远程仓库"形式展示
4. 浏览过的图片自动缓存到本地
5. 缩略图异步生成并缓存
```

## 三、场景B/C：手机 ↔ 手机 / 手机 ↔ 平板

### 3.1 方案对比

| 方案 | 复杂度 | 性能 | 发现机制 | 跨平台 | 推荐度 |
|------|--------|------|---------|--------|--------|
| **HTTP Server + mDNS** | ⭐⭐ 低 | ⭐⭐⭐⭐ | mDNS 自动发现 | ✅ | 🏆 **首选** |
| FTP Server | ⭐⭐ 低 | ⭐⭐⭐ | 手动输入 IP | ✅ | 备选 |
| WebDAV Server | ⭐⭐⭐ 中 | ⭐⭐⭐ | 手动输入 IP | ✅ | 备选 |
| WiFi Direct | ⭐⭐⭐⭐ 高 | ⭐⭐ | 系统级发现 | ❌ 仅 Android | ❌ 不推荐 |
| Bluetooth | ⭐⭐⭐ 中 | ⭐ | 系统级发现 | ❌ 极慢 | ❌ 不推荐 |
| WebRTC P2P | ⭐⭐⭐⭐⭐ 高 | ⭐⭐⭐⭐ | STUN/TURN | ✅ | 远期考虑 |

### 3.2 推荐方案：HTTP Server + mDNS 自动发现

**架构示意：**

```
┌─────────────────────┐          ┌─────────────────────┐
│   设备 A（服务端）    │          │   设备 B（客户端）    │
│                     │          │                     │
│  remoPhoto App      │          │  remoPhoto App      │
│  ┌───────────────┐  │          │  ┌───────────────┐  │
│  │ ImageServer   │  │  HTTP   │  │ RemoteRepo    │  │
│  │ (NanoHTTPd)   │◄─┼─────────┼─►│ Client        │  │
│  │ :8080         │  │          │  │               │  │
│  └───────────────┘  │          │  └───────────────┘  │
│  ┌───────────────┐  │          │  ┌───────────────┐  │
│  │ mDNS Service  │  │ 局域网    │  │ mDNS Discovery│  │
│  │ "remoPhoto-   │  │ 自动发现  │  │ 扫描局域网设备  │  │
│  │  张三的手机"   │  │          │  │               │  │
│  └───────────────┘  │          │  └───────────────┘  │
└─────────────────────┘          └─────────────────────┘
```

**工作流程：**

```
服务端（提供图片的一方）：
1. 用户在设置中开启"允许远程访问"
2. App 启动内嵌 HTTP Server（NanoHTTPd，端口 8080）
3. 注册 mDNS 服务（服务名: "remoPhoto-{设备名}"）
4. HTTP API 提供：
   - GET /api/albums          → 相册列表（JSON）
   - GET /api/album/{id}/images → 图片列表（JSON）
   - GET /api/image/{id}      → 图片文件流
   - GET /api/image/{id}/thumb → 缩略图
   - GET /api/image/{id}/info  → 图片元数据

客户端（浏览图片的一方）：
1. 用户在设置中进入"添加远程仓库"
2. App 通过 mDNS 扫描局域网内的 remoPhoto 服务
3. 显示可用设备列表（如"张三的手机"、"客厅平板"）
4. 选择目标设备，建立连接
5. 远程仓库数据以与本地仓库相同的方式展示
6. 图片加载时带网络进度指示
```

### 3.3 为什么 HTTP + mDNS 是最优选择？

| 优势 | 说明 |
|------|------|
| **零配置** | mDNS 自动发现，用户无需输入 IP 地址 |
| **轻量级** | NanoHTTPd 仅 ~100KB，不增加 APK 体积负担 |
| **高性能** | HTTP 支持 Range 请求，大图可分段加载；支持 Keep-Alive 复用连接 |
| **易调试** | HTTP 协议标准，可用浏览器直接测试 API |
| **安全性** | 仅限局域网访问，不暴露到公网；可加 Token 认证 |
| **扩展性** | 未来可轻松扩展为 WebDAV 或增加 WebSocket 实时通知 |

### 3.5 备选方案：FTP Server

如果 HTTP Server 方案无法满足某些场景，可选择 FTP：

**优点：**
- Apache FTPServer 等成熟库可直接用
- 支持完整的目录操作
- 客户端可用现有 FTP 库

**缺点：**
- 无自动发现机制，需手动输入 IP 和端口
- 主动模式/被动模式配置复杂（NAT 穿透）
- 用户体验不如 mDNS + HTTP 直观

## 四、安全设计

### 4.1 局域网隔离

- 远程服务**仅监听局域网 IP**（如 `192.168.x.x`），不绑定 `0.0.0.0`
- 不通过 UPnP 自动端口映射到公网

### 4.2 访问控制

```
┌────────────────────────────────────────┐
│            安全层级                      │
│                                        │
│  Layer 1: 局域网隔离（仅 WiFi 可访问）    │
│  Layer 2: 用户手动开启/关闭远程服务       │
│  Layer 3: 可选 Token/密码认证            │
│  Layer 4: 只读模式（不暴露写入接口）      │
└────────────────────────────────────────┘
```

### 4.3 隐私保护

- 服务端可设置"仅共享指定相册"（不是全部目录）
- 客户端仅读取，无删除/修改权限
- 服务端可查看"当前连接的设备"列表
- 空闲 5 分钟后自动断开连接

## 五、缓存策略

```
本地缓存层级：
┌──────────────────────────────────────┐
│ L1: 内存缓存（LRU, 128MB）           │  ← 最快，当前浏览的图片
├──────────────────────────────────────┤
│ L2: 磁盘缩略图缓存（LRU, 200MB）      │  ← 缩略图持久化
├──────────────────────────────────────┤
│ L3: 磁盘原图缓存（LRU, 500MB）        │  ← 最近浏览的原图
└──────────────────────────────────────┘

缓存淘汰策略：
- L1: 最近最少使用，全屏切换时触发回收
- L2: 存储空间不足时淘汰，优先保留当前相册
- L3: 总大小超限后淘汰最旧文件，保留最近访问
- 用户可手动清除所有远程缓存
- **离线降级**：网络不可用时自动回退到本地缓存的低清版本，网络恢复后切换到高清
```

### `/api/albums` 当前兼容字段（2026-06-22）

```json
{
  "albums": [
    {
      "id": 42,
      "name": "示例相册",
      "imageCount": 120,
      "repositoryId": 1,
      "parentAlbumId": null,
      "coverImageId": 1001,
      "lastModified": 1781010155000
    }
  ]
}
```

- `coverImageId`、`lastModified` 均为向后兼容的可选字段。
- 客户端连接旧服务端时，缺失字段分别按 `null`、`0` 处理。
- 服务端只暴露本地仓库，禁止将已同步远程仓库再次中继。
- 大型列表使用 Room 元数据缓存；协议同步成功后更新仓库 `last_scan_time`，客户端以 30 分钟 TTL 判定是否刷新。
- 具体契约和兼容测试见 [测试用例.md](./测试用例.md)。

## 七、总结建议

```
推荐实施路线：

第一优先级：SMB 协议（场景A：电脑 → 手机）
  ├── 技术成熟、配置简单、用户基数大
  ├── 库：com.hierynomus:smbj
  └── 预计工作量：2-3 周

第二优先级：HTTP Server + mDNS（场景B/C：设备间）
  ├── 轻量级、零配置、用户体验好
  ├── 库：NanoHTTPd + JmDNS
  └── 预计工作量：3-4 周

远期考虑（视需求）：
  └── WebRTC P2P（跨网络访问、NAT 穿透）
      └── 技术复杂、仅当用户有跨局域网需求时再启动
```
