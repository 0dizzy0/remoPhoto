## Phase 2 新增问题修复

| ID | 问题 | 状态 | 修复说明 |
|----|------|------|---------|
| P2-F01 | 全屏闪退：FocusRequester 未初始化 | ✅ | `LaunchedEffect(animVisible)` + `delay(300L)` 等 AnimatedVisibility 动画完成后再请求焦点 |
| P2-F02 | 自定义排序后全屏图片错位 | ✅ | FullScreenViewModel 同步读取相册/全局排序，与画廊保持一致顺序 |
| P2-F03 | 画廊图片列表不加载（Flow collect 阻塞） | ✅ | `settingsRepository.defaultSortOrder.collect` 改为 `.first()`，避免永不返回的挂起 |
| P2-F04 | 相册标签 toggle 会清除所有标签 | ✅ | `CategoryDao.deleteCrossRef(albumId, categoryId)` 精确删除，替代 deleteCrossRefsByAlbum |
| P2-F05 | 分类筛选未生效：点击分类后相册未筛选 | ✅ | AlbumListViewModel 增加 `loadAlbumsByCategory()` + 树形筛选；NavGraph 路由支持 `album_list?categoryId=X&categoryName=XX` 查询参数 |
| P2-F06 | 缩略图播放 GIF 动图影响流畅度 | ✅ | ImageLoaderFactory 新增 `createThumbnailLoader()` 不含动图解码器；ImageThumbnail 使用独立 loader |
| P2-F07 | 分类筛选状态残留：回退后仍显示筛选 | ✅ | `LaunchedEffect(categoryId)` 增加 else 分支，`categoryId==null` 时调用 `clearCategoryFilter()` |

### 详细分析

**P2-F01: FocusRequester is not initialized**
- 堆栈: `FullScreenViewer.kt:235` → `FocusRequester.requestFocus()`
- 原因: `LaunchedEffect(Unit)` 在 AnimatedVisibility 进场动画（250ms tween）完成前就请求焦点，focus target node 尚未就绪
- 修复: 延迟 300ms + try-catch 保护

**P2-F02: 排序后缩略图→全屏图片不正确**
- 原因: GalleryViewModel 和 FullScreenViewModel 各自独立加载图片，只有 Gallery 应用了排序，Viewer 始终用 `last_modified ASC` 数据库默认排序
- 修复: FullScreenViewModel.loadImages() 同步读取相册 sortOrder + 全局 defaultSortOrder，应用相同排序

**P2-F03: 画廊图片列表不加载**
- 原因: `settingsRepository.defaultSortOrder.collect {}` 是无限挂起的 Flow，后续 `imageRepository.getImagesByAlbum().collect {}` 代码永远无法执行
- 修复: 改为 `.first()` 只取当前值

**P2-F04: 相册标签 toggle 一次全部清空**
- 原因: `CategoryManager.removeAlbumFromCategory()` 调用 `deleteCrossRefsByAlbum(albumId)`，删除了该相册的所有关联
- 修复: CategoryDao 新增 `deleteCrossRef(albumId, categoryId)` 精确删除单条关联

**P2-F05: 分类筛选未生效**
- 原因: `CategoryListScreen.onCategoryClick` 仅 `popBackStack()`，相册列表无筛选逻辑
- 修复: NavGraph 路由升级为 `album_list?categoryId={}&categoryName={}` 带查询参数；AlbumListViewModel 新增 `loadAlbumsByCategory()` + `filterTreeByCategory()` 树形筛选

**P2-F06: 缩略图播放 GIF 动图影响流畅度**
- 原因: 全局 Coil ImageLoader 含 GIF/ImageDecoder 解码器，缩略图也解码动画帧
- 修复: `ImageLoaderFactory.createThumbnailLoader()` 创建不含动图解码器的独立 loader；`ImageThumbnail` 使用 `thumbnailImageLoader`

**P2-F07: 分类筛选状态残留**
- 原因: 共享 `albumListViewModel`，进入分类筛选后 `_filterCategoryId` 被设置，返回普通列表时 `LaunchedEffect(categoryId==null)` 未清理
- 修复: else 分支调用 `clearCategoryFilter()`，无筛选参数时重置 ViewModel 状态

## Phase 3 验证问题修复

> 最后更新: 2026-06-10（Phase 3 收敛）

| ID | 问题 | 状态 | 修复说明 |
|----|------|:----:|---------|
| P3-F01 | 页面转场动画方向不正确：底部 Tab 切换也有 slide 动画 | ✅ | NavHost 层级 slide 动画移除，Tab 切换改为即时切换；Navigation Compose 新 API 的 `enterTransition` 优先级高于 `navOptions.anim{}` |
| P3-F02 | 数据库导出文件名与 Room 实际文件名不匹配 | ✅ | DatabaseExporter `DB_NAME` 从 `remophoto_db` 修正为 `remophoto.db`（与 AppDatabase.DATABASE_NAME 一致） |
| P3-F03 | 数据库导入后相册封面丢失 | ✅ | 导入后不再删除 WAL/SHM 文件（Room WAL 模式，未 checkpoint 数据仅在 WAL 中） |
| P3-F04 | 数据库导入后 Room 实例未关闭，旧数据残留内存 | ✅ | 导入前调用 `AppDatabase.closeInstance()` 释放文件锁；导入后 `Process.killProcess()` 重启进程加载新数据库 |

### 详细分析

**P3-F01: 底部 Tab 切换有 slide 动画**
- 原因: Navigation Compose 2.8.x 的 `NavHost(enterTransition=...)` 对所有目标生效，优先级高于 `navigate()` 中的 `navOptions { anim { enter=0 } }`
- 日志: `📱 底部Tab点击: 设置 (from=album_list...)` → 路由切换正确但动画异常
- 修复: 移除 NavHost 层级 `enterTransition`/`exitTransition`/`popEnterTransition`/`popExitTransition`，底部 Tab 切换即时无动画；深层导航内部使用 AnimatedContent/AnimatedVisibility 提供视觉过渡

**P3-F02: 导出文件名不匹配**
- 原因: `DatabaseExporter.DB_NAME = "remophoto_db"`，但 Room 实际文件名为 `remophoto.db`
- 修复: 统一为 `remophoto.db`

**P3-F03: 导入后封面丢失**
- 原因: Room 使用 WAL 模式，封面路径更新写入 `.db-wal` 未 checkpoint 到 `.db`；导入时打包了 WAL 但之后删除了它
- 修复: 保留 WAL/SHM 文件不删除，让 Room 重启后自动恢复

**P3-F04: 导入后需重启**
- 原因: Room 单例持有旧数据库引用，导入文件后未重新打开
- 修复: 导入前 `closeInstance()` 关闭单例，导入后 `Process.killProcess()` 重启进程

> 最后更新: 2026-06-10

| ID | 问题 | 状态 | 说明 |
|----|------|:----:|------|
| P3-F10a | 相册标题过大 | ✅ | TopAppBar title `titleLarge`(22sp) → `titleSmall`(14sp) |
| P3-F10b | 全屏进出 scale 动画跳跃 | ✅ | `scaleIn/Out`(0.85↔1.0) → 纯 `fadeIn/fadeOut` |
| P3-F10c | Scaffold 响应 window insets 跳动 | ✅ | Scaffold → Box 布局；TopAppBar `windowInsets = WindowInsets(0)` |
| P3-F10d | FLAG_HIDE_NAVIGATION 底部跳动 | ✅ | 移除 `FLAG_HIDE_NAVIGATION`，仅保留 `FLAG_FULLSCREEN` |
| P3-F10e | FLAG_FULLSCREEN 窗口尺寸变化 | ✅ | 彻底移除所有 `systemUiVisibility` 修改，仅依赖 `enableEdgeToEdge()` |

**根因分析（7 轮日志追踪）：**

1. TopAppBar 高度 192px→300px：`TopAppBarDefaults.windowInsets` 含状态栏 → 加 `windowInsets=0`
2. 内容区 108px 跳动：`FLAG_FULLSCREEN` 导致系统释放状态栏 36dp 窗口空间 → 外层 Box 2300→2408
3. 结论：`FLAG_FULLSCREEN` 必然导致 Android 窗口尺寸变化（`LAYOUT_STABLE` 在 Vivo 无效）

**最终方案：** 完全移除 `systemUiVisibility` 修改，状态栏保持透明 edge-to-edge
