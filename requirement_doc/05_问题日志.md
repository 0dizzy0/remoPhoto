## Phase 4 验证问题（2026-06-12）

### P4-F01: DB Migration FK 约束导致启动闪退

- **现象:** App 启动时 Room schema 校验失败，`IllegalStateException: Migration didn't properly handle`
- **根因:** MIGRATION_2_3 中 `ALTER TABLE ... REFERENCES remote_connections(id) ON DELETE SET NULL` 在 SQL 层面创建了 FK 约束，但 `RepositoryEntity` 未声明 `@ForeignKey`，导致 Expected vs Found 不匹配
- **解决:** 移除 SQL 中的 `REFERENCES` 子句，FK 关系由应用层 DAO 维护（与 `ImageEntity.albumId` 等字段做法一致）
- **影响:** MIGRATION_2_3 仅添加列，不再创建 FK

### P4-F02: ForegroundService 超时导致设置页闪退

- **现象:** 点击"允许远程访问"开关后闪退，`ForegroundServiceDidNotStartInTimeException`
- **根因:** `startForeground()` 必须在 `startForegroundService()` 后 5s 内调用，但原实现先同步启动 HTTP Server（含 LAN IP 检测），耗时可能超过 5s
- **解决:** 先立即调用 `startForeground()` 显示"正在启动..."通知，改为 `MainScope().launch` 后台异步启动 Server，成功后更新通知内容
- **影响:** HttpServerForegroundService 重写为异步启动模式

### P4-F03: 旧 DB 残留导致升级后闪退

- **现象:** 首次修复安装后仍然崩溃（因为 MIGRATION_2_3 已在手机上执行过，留下了带 FK 的 schema）
- **解决:** 开发环境执行 `adb uninstall` 清理残留，重新安装后 DB 全新创建
- **影响:** 开发阶段一次性处理，生产环境需添加 `fallbackToDestructiveMigration()` 或创建修复迁移

### P4-F04: 空仓库列表时无法添加远程仓库

- **现象:** 主界面在未添加任何本地仓库前，只有"暂无仓库"的空状态提示（含"添加仓库"按钮跳转仓库管理页），远程仓库的"＋ 添加"按钮不可见
- **根因:** `AlbumListScreen.kt` 中仓库列表视图通过 `if (allRepos.isEmpty())` 提前返回 `EmptyStateView`，远程仓库区域（含"＋ 添加"按钮）位于 `else` 分支内
- **解决:** 移除 `allRepos.isEmpty()` 提前返回，始终渲染 `LazyColumn`。本地仓库为空时显示"暂无本地仓库"提示 + "＋ 添加本地仓库"快捷按钮；远程仓库区域始终可见
- **影响:** `AlbumListScreen.kt` 仓库列表视图重构

### P4-F05: mDNS 服务注册从未被调用

- **现象:** 开启"允许远程访问"后，同局域网另一设备无法通过 mDNS 发现本设备；`adb logcat` 无任何 mDNS 注册日志
- **根因:** `HttpServerForegroundService.startServer()` 仅创建 `HttpServerManager` 启动 HTTP Server，但**从未创建 `MdnsRegistrar` 或调用其 `register()` 方法**；`WifiLockManager`（MulticastLock）也未获取
- **解决:** 在 `HttpServerForegroundService` 中添加 `MdnsRegistrar` 和 `WifiLockManager` 字段。HTTP Server 启动成功后：获取 MulticastLock + WifiLock → 注册 mDNS `_remophoto._tcp` 服务。Service 停止时：注销 mDNS → 释放锁
- **影响:** `HttpServerForegroundService.kt` 增加 mDNS 生命周期管理

### P4-F06: HTTP Server 缺少 /api ping 端点

- **现象:** 手动输入 IP:端口添加远程设备时，点击"测试连接"始终返回"连接失败"
- **根因:** `RemoteHttpClient.ping()` 请求 `GET /api`，但 `HttpServerManager` 的 `serve()` 路由表中没有 `/api` 端点，返回 404 Not Found
- **解决:** 添加 `/api` → `handlePing()` 路由，返回 `{"status":"ok","version":"1.0.0","app":"remoPhoto"}`
- **影响:** `HttpServerManager.kt` 新增健康检查端点

### P4-F07: 远程同步后 loadAlbums() 自取消导致数据无法显示

- **现象:** 发现并连接远程设备后，点击远程仓库无法看到相册列表（一直加载中或显示空）
- **根因:** `syncAndLoadRemoteAlbums()` 调用 `loadAlbums()`，而 `loadAlbums()` 首行 `loadJob?.cancel()` 取消了正在执行同步的协程自身，导致数据加载竞态
- **解决:** `syncAndLoadRemoteAlbums()` 改为内联相册加载逻辑（直接调用 `getRootAlbumsByRepository` + `buildAlbumTree`），不再调用 `loadAlbums()`
- **影响:** `AlbumListViewModel.kt` syncAndLoadRemoteAlbums 方法重构

### P4-F08: 远程图片元数据从未同步到本地 DB

- **现象:** 能看到远程相册列表（修复 P4-F07 后），但点击进入相册看不到图片
- **根因:** `syncAndLoadRemoteAlbums` 只调用了 `syncAlbums()`（同步相册元数据），**从未调用 `syncImages()`**（同步图片元数据）。`GalleryViewModel` 查本地 DB 时图片表为空
- **解决:** `GalleryViewModel.loadAlbumAndImages()` 增加远程相册检测：若 `repository.remoteConnectionId != null`，先调用 `syncImages()` 按需同步当前相册图片到本地 DB，再通过 Flow 加载
- **影响:** `GalleryViewModel.kt` 新增远程同步依赖（RemoteConnectionDao, RepositoryDao, SyncRemoteRepositoryUseCase）

### P4-F09: 默认 ImageLoader 未注册 RemoteImageFetcher

- **现象:** 图片元数据同步到本地 DB 后，Gallery 网格和全屏查看器中远程图片不显示（Coil 无法加载 `http://` URL）
- **根因:** `ImageLoaderFactory.create()` 默认 loader 的 `components {}` 只注册了 `GifDecoder.Factory()` 和 `ImageDecoderDecoder.Factory()`，未注册 `RemoteImageFetcher.Factory()`。远程图片 `http://` URL 无对应 Fetcher 处理
- **解决:** 在默认 `create()` 和 `createThumbnailLoader()` 的 `components {}` 中添加 `add(RemoteImageFetcher.Factory())`
- **影响:** `ImageLoader.kt` 默认 loader 增加远程图片加载能力

### P4-F10: mDNS 发现/注册失败（权限 + IP 获取缺陷）

- **现象:** 修复 P4-F05~F09 后，再次测试时无法发现局域网内的远程设备（之前能发现）
- **根因:** 两个独立缺陷叠加：
  1. `AndroidManifest.xml` 缺少 `CHANGE_WIFI_MULTICAST_STATE` 权限 → `MulticastLock.acquire()` 静默失败 → mDNS 多播包无法收发
  2. `MdnsRegistrar.getLanIp()` 和 `WifiLockManager.getLanIp()` 仅使用 `wifiManager.connectionInfo.ipAddress`（Android 10+ 因隐私限制常返回 0），缺少 `NetworkInterface` 枚举回退。巧合的是 `HttpServerManager.getLanIpAddress()` 有回退 → HTTP Server 启动成功 + 通知显示，但 mDNS 注册/发现因获取不到 IP 而跳过
- **解决:** 
  1. `AndroidManifest.xml` 添加 `CHANGE_WIFI_MULTICAST_STATE` 权限
  2. `MdnsRegistrar.getLanIp()` 和 `WifiLockManager.getLanIp()` 增加 `NetworkInterface.getNetworkInterfaces()` 枚举回退（与 HttpServerManager 一致）
- **影响:** `AndroidManifest.xml`, `MdnsRegistrar.kt`, `WifiLockManager.kt`

### P4-F11: JmDNS NetworkOnMainThreadException + HTTP 明文被拦截

- **现象:** 添加远程仓库对话框打开后，mDNS 发现每次启动即崩溃（`start()` 返回 `false`，`isScanning` 始终 `false`）；手动输入 IP 测试连接报 `Cleartext HTTP traffic not permitted`
- **根因:** 两个独立问题：
  1. `JmDNS.create()` 内部调用 `InetAddress.getLocalHost()` 是网络操作，从 `LaunchedEffect`（主线程）调用触发 Android StrictMode `NetworkOnMainThreadException`。`MdnsDiscoveryService.start()` 未使用 `Dispatchers.IO`
  2. Android 9+ 默认禁止 HTTP 明文流量，`Application` 未声明 `usesCleartextTraffic="true"`
- **解决:** 
  1. `MdnsDiscoveryService.start()`/`stop()` 改为 `suspend fun`，JmDNS 创建/关闭通过 `withContext(Dispatchers.IO)` 在 IO 线程执行
  2. `AndroidManifest.xml` 添加 `android:usesCleartextTraffic="true"`
- **影响:** `MdnsDiscoveryService.kt`（suspend 化）, `AddRemoteRepoDialog.kt`（协程适配）, `AndroidManifest.xml`

### P4-F12: mDNS 发现自身 + 远程删除牵连本地 + 远程仓库无操作入口

- **现象:** 
  1. 两台设备同时开启远程访问时，mDNS 发现自己导致混淆
  2. 删除远程仓库时本地仓库数据也被清理
  3. 无本地仓库时仓库管理页无远程添加/同步入口
- **根因:** 
  1. `MdnsDiscoveryService.emitDevices()` 未过滤本机 LAN IP
  2. `RepositoryManager.deleteRepository()` 未区分远程/本地仓库，删除逻辑一致但用户感知混乱（实际不会删除本地，但缺少远程关联清理）
  3. `RepositoryManagerScreen` 未区分远程/本地仓库，远程仓库显示扫描按钮但无意义，且无远程仓库直接添加入口
- **解决:** 
  1. `emitDevices()` 保存 `localIp`，过滤 `device.host == localIp` 的设备
  2. `deleteRepository()` 增加 `remoteConnectionId` 检测 → 清理 `RemoteConnectionEntity` + `KeyStoreManager.deleteCredential()`
  3. `RepositoryItem` 卡片区分远程(🌐)/本地(📁)，远程隐藏扫描/暂停按钮，删除确认对话框区分文案
- **影响:** `MdnsDiscoveryService.kt`, `RepositoryManagerViewModel.kt`, `RepositoryManagerScreen.kt`

### P4-F13: 远程仓库添加成功但主界面不显示

- **现象:** mDNS 发现设备后点击连接提示"已添加"，手动添加测试成功、提示成功，但主界面远程仓库区域始终为空
- **根因:** `addRemoteRepo()` 中 `RemoteConnectionEntity` 插入（`connDao.insert`）和 `RepositoryEntity` 插入（`repoDao.insert`）非原子操作。连接记录写入成功但仓库记录写入失败时，DB 残留孤儿连接。再次添加时 `getConnectionByHostAndPort` 命中已存在的连接 → 提示"已添加"，但无对应仓库 → 主界面 `remoteRepos()` 过滤无结果。形成死循环：提示已添加 + 永远不显示
- **解决:** 1) 连接已存在时检查关联仓库是否存在 → 不存在则补建 2) 仓库插入失败时回滚删除连接记录 3) 全链路增加数据一致性日志
- **影响:** `AddRemoteRepoDialog.kt` addRemoteRepo 函数重写

### P4-F14: 服务后台断开 + 自身可添加 + 设备名 UI 缺陷 + 名称不持久化

- **现象:** 1) 切出设置页面后远程服务自动断开 2) 仍可添加本机自身为远程仓库 3) 开启服务后设备名输入框消失 4) 关闭开关/退出后设备名丢失
- **根因:** 
  1) `toggleServer()` 未同步调用 `setHttpServerEnabled()` → DataStore 中 `http_server_enabled` 始终为 `false` → ViewModel 重建后无法自动恢复服务
  2) `addRemoteRepo()` 缺少本机 IP 校验
  3) `if (!serverRunning)` 条件包裹导致 `OutlinedTextField` 直接移除
  4) `remember { mutableStateOf(deviceName) }` 不同步 DataStore 变更；保存逻辑依赖 UI 层 `editDeviceName != deviceName` 判断
- **解决:** 
  1) `toggleServer` 增加 `setHttpServerEnabled()`；`init` 中检测 `httpServerEnabled==true` → 自动 `startServer()`
  2) `addRemoteRepo` 添加 `WifiLockManager.getLanIp()` 比对，`host==myIp` 则拒绝
  3) 移除 `if` 包裹 → `enabled = !serverRunning` + `supportingText = "停止服务后可修改"`
  4) `LaunchedEffect(deviceName)` 同步 `editDeviceName`；DataStore `setDeviceName` 确认正常
- **影响:** `SettingsViewModel.kt`, `SettingsScreen.kt`, `AddRemoteRepoDialog.kt`

### 详细分析

**P2-F01: FocusRequester is not initialized**
- 堆栈: `FullScreenViewer.kt:235` → `FocusRequester.requestFocus()`
- 原因: `LaunchedEffect(Unit)` 在 AnimatedVisibility 进场动画（250ms tween）完成前就请求焦点，focus target node 尚未就绪
- 修复: 延迟 300ms + try-catch 保护

**P2-F02: 排序后缩略图→全屏图片不正确**
- 原因: GalleryViewModel 和 FullScreenViewModel 各自独立加载图片，只有 Gallery 应用了排序，Viewer 始终用 `last_modified ASC` 数据库默认排序
- 修复: FullScreenViewModel.loadImages() 同步读取相册 sortOrder + 全局 defaultSortOrder，应用相同排序

**P2-F03: 画廊图片列表不加载**
- 原因: `settingsRepository.defaultSortOrder.collect {}` 是无限挂起的 Flow，后续 `imageRepository.getImagesByAlbum().collect {}` 代码永远无法执行
- 修复: 改为 `.first()` 只取当前值

**P2-F04: 相册标签 toggle 一次全部清空**
- 原因: `CategoryManager.removeAlbumFromCategory()` 调用 `deleteCrossRefsByAlbum(albumId)`，删除了该相册的所有关联
- 修复: CategoryDao 新增 `deleteCrossRef(albumId, categoryId)` 精确删除单条关联

**P2-F05: 分类筛选未生效**
- 原因: `CategoryListScreen.onCategoryClick` 仅 `popBackStack()`，相册列表无筛选逻辑
- 修复: NavGraph 路由升级为 `album_list?categoryId={}&categoryName={}` 带查询参数；AlbumListViewModel 新增 `loadAlbumsByCategory()` + `filterTreeByCategory()` 树形筛选

**P2-F06: 缩略图播放 GIF 动图影响流畅度**
- 原因: 全局 Coil ImageLoader 含 GIF/ImageDecoder 解码器，缩略图也解码动画帧
- 修复: `ImageLoaderFactory.createThumbnailLoader()` 创建不含动图解码器的独立 loader；`ImageThumbnail` 使用 `thumbnailImageLoader`

**P2-F07: 分类筛选状态残留**
- 原因: 共享 `albumListViewModel`，进入分类筛选后 `_filterCategoryId` 被设置，返回普通列表时 `LaunchedEffect(categoryId==null)` 未清理
- 修复: else 分支调用 `clearCategoryFilter()`，无筛选参数时重置 ViewModel 状态

## Phase 3 验证问题修复

> 最后更新: 2026-06-10（Phase 3 收敛）

| ID | 问题 | 状态 | 修复说明 |
|----|------|:----:|---------|
| P3-F01 | 页面转场动画方向不正确：底部 Tab 切换也有 slide 动画 | ✅ | NavHost 层级 slide 动画移除，Tab 切换改为即时切换；Navigation Compose 新 API 的 `enterTransition` 优先级高于 `navOptions.anim{}` |
| P3-F02 | 数据库导出文件名与 Room 实际文件名不匹配 | ✅ | DatabaseExporter `DB_NAME` 从 `remophoto_db` 修正为 `remophoto.db`（与 AppDatabase.DATABASE_NAME 一致） |
| P3-F03 | 数据库导入后相册封面丢失 | ✅ | 导入后不再删除 WAL/SHM 文件（Room WAL 模式，未 checkpoint 数据仅在 WAL 中） |
| P3-F04 | 数据库导入后 Room 实例未关闭，旧数据残留内存 | ✅ | 导入前调用 `AppDatabase.closeInstance()` 释放文件锁；导入后 `Process.killProcess()` 重启进程加载新数据库 |

### 详细分析

**P3-F01: 底部 Tab 切换有 slide 动画**
- 原因: Navigation Compose 2.8.x 的 `NavHost(enterTransition=...)` 对所有目标生效，优先级高于 `navigate()` 中的 `navOptions { anim { enter=0 } }`
- 日志: `📱 底部Tab点击: 设置 (from=album_list...)` → 路由切换正确但动画异常
- 修复: 移除 NavHost 层级 `enterTransition`/`exitTransition`/`popEnterTransition`/`popExitTransition`，底部 Tab 切换即时无动画；深层导航内部使用 AnimatedContent/AnimatedVisibility 提供视觉过渡

**P3-F02: 导出文件名不匹配**
- 原因: `DatabaseExporter.DB_NAME = "remophoto_db"`，但 Room 实际文件名为 `remophoto.db`
- 修复: 统一为 `remophoto.db`

**P3-F03: 导入后封面丢失**
- 原因: Room 使用 WAL 模式，封面路径更新写入 `.db-wal` 未 checkpoint 到 `.db`；导入时打包了 WAL 但之后删除了它
- 修复: 保留 WAL/SHM 文件不删除，让 Room 重启后自动恢复

**P3-F04: 导入后需重启**
- 原因: Room 单例持有旧数据库引用，导入文件后未重新打开
- 修复: 导入前 `closeInstance()` 关闭单例，导入后 `Process.killProcess()` 重启进程
