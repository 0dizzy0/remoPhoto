# Phase 4 性能优化与问题修复方案

更新时间：2026-06-21

实施状态更新：2026-06-22

> 当前已完成数据库一致性备份、后台远程服务、远程封面、滚轮事件、SAF 单游标扫描、磁盘 Spool、前台 WorkManager、事务索引切换和阶段化数量 UI。目录队列级持久化检查点与真实大仓库完整性能对比仍未完成，详见本文末尾“实施结果”。

## 1. 背景与当前状态

Phase 4 远程仓库基础功能已经完成，前一轮已处理自动发现、远程服务重复启动、远程数据循环共享以及设置页切换卡顿等问题。目前待处理以下五项：

1. 数据库导入导出需要包含远程仓库信息，并兼容旧版本备份。
2. 手机息屏或应用进入后台后，远程仓库可能无法再次连接。
3. 远程仓库的相册封面无法显示。
4. 全屏浏览图片时，蓝牙鼠标滚轮无法翻页。
5. 数十万张图片、100GB 以上本地仓库扫描速度慢，长任务 UI 和任务恢复能力不足。

实施时不得大幅调整现有 MVVM / Repository / UseCase 架构。关键阶段必须写入 `AppLogger` 日志。

## 2. 问题一：数据库导入导出与向下兼容

### 已确认根因

- 当前实现直接复制正在使用的 Room 主数据库以及 WAL/SHM，导出期间继续写入时可能产生不一致快照。
- 旧导入逻辑将 ZIP 中的 `settings/settings.preferences_pb` 解压到数据库目录，但没有可靠创建父目录。
- 导入后通过已经关闭的 Room DAO 修改远程连接状态，可能失败。
- 备份没有明确的格式版本、数据库版本与功能清单，无法可靠判断未来版本备份是否可导入。

### 实施方案

- 在 `AppDatabase` 暴露当前 `SCHEMA_VERSION`。
- 导出时使用 SQLite `VACUUM INTO` 创建一致性数据库快照，不再直接打包活动 WAL/SHM。
- ZIP 增加 `manifest.json`，至少包含：
  - `formatVersion`
  - `databaseVersion`
  - `createdAt`
  - `includesRemoteConnections`
  - `remoteConnectionCount`
- 远程连接元信息继续随 Room 数据库导出，包括协议、IP、端口、设备名、共享名和用户名；Android Keystore 中的密码或 Token 不导出。
- 导入时先解压到应用缓存临时目录，禁止路径穿越，只接受数据库、WAL/SHM、DataStore 和 manifest 等已知文件。
- 兼容以下输入：
  - 新版带 manifest 的 ZIP；
  - 旧版 DB + WAL/SHM ZIP；
  - 不含 DataStore 的旧 ZIP；
  - 旧版原始 `.db` 文件。
- 导入前执行 SQLite 完整性检查并读取 `user_version`：
  - 版本 1～当前版本：允许导入，重启后由 Room Migration 升级；
  - 高于当前版本：拒绝导入并提示升级应用；
  - 非法或损坏数据库：拒绝导入。
- 导入数据库中的远程连接状态重置为 `DISCONNECTED`，但保留连接元信息；跨设备凭据由用户重新输入。
- 替换前为当前数据库创建一致性备份，任一环节失败时回滚。

### 关键日志

- 导出格式版本、数据库版本、远程连接数量、是否包含设置。
- 导入来源类型、检测到的数据库版本、完整性校验结果。
- 远程连接保留数量、状态重置数量、迁移目标版本。
- 回滚开始、成功或失败原因。

## 3. 问题二：息屏与后台后的远程连接自愈

### 已确认根因

- 当前仅持有 `WifiLock` 和 `MulticastLock`，没有 CPU `PARTIAL_WAKE_LOCK`；深度休眠时 HTTP 接收线程可能无法及时运行。
- 应用进程仍存在但前台服务被厂商系统终止时，`Application.onCreate()` 不会再次执行，回到前台缺少恢复入口。
- Wi-Fi 重连或 IP 变化后，HTTP Server 和 JmDNS 仍可能绑定旧地址。

### 实施方案

- 扩展 `WifiLockManager`，增加 `PowerManager.PARTIAL_WAKE_LOCK`；仅在“允许远程访问”启用且服务运行时持有，停止服务立即释放。
- 将“确保远程服务运行”提取为 Application 级幂等方法；`MainActivity.onResume()` 触发一次自愈检查。
- 前台服务监听默认网络变化及屏幕唤醒：
  - IP 未变化：重新注册 mDNS；
  - IP 变化或 HTTP Server 已停止：重新绑定 HTTP Server，并重新注册 mDNS；
  - 使用互斥锁避免多个网络回调并发重启。
- 保留已有重复启动保护，避免再次出现 `EADDRINUSE`。
- 不自动请求忽略电池优化；如特定厂商仍强杀服务，只在 UI 中提供说明，不擅自修改系统设置。

### 关键日志

- WakeLock、WifiLock、MulticastLock 的获取与释放。
- App 回前台自愈检查结果。
- 网络可用、丢失、IP 变化、mDNS 重注册和 HTTP 重绑定原因。

## 4. 问题三：远程相册封面

### 已确认根因

- `/api/albums` 仅返回相册基本信息，没有封面图片 ID。
- `SyncRemoteRepositoryUseCase` 创建远程相册时将 `coverImagePath` 固定为 `null`。

### 实施方案

- 服务端 `/api/albums` 为每个相册增加可选字段 `coverImageId`：
  - 优先使用相册自定义封面对应的图片 ID；
  - 没有自定义封面时使用按现有规则选出的第一张图片；
  - 空相册返回 `null`。
- `RemoteAlbumDto` 以可选字段解析 `coverImageId`，确保新客户端仍可连接旧服务端。
- 同步远程相册时将封面转换为：
  `http://host:port/api/image/{coverImageId}/thumb`
- 旧客户端会忽略额外 JSON 字段，因此服务端协议保持向后兼容。
- 避免新增固定 `/album/{id}/cover` 缓存地址；使用图片 ID 可在封面改变后自然生成新的 Coil 缓存键。

### 验收

- 远程相册列表的网格和列表模式均能显示封面。
- 空相册继续显示占位图。
- 连接旧版服务端时不崩溃，只是没有封面。

## 5. 问题四：蓝牙鼠标滚轮翻页

### 已确认根因

`ZoomableImage` 的滚轮判断位于 `awaitFirstDown()` 之后，但鼠标 `ACTION_SCROLL` 不会先产生触摸按下事件，所以该分支通常无法执行。

### 实施方案

- 增加独立的 Pointer Scroll 监听循环，不依赖 `awaitFirstDown()`。
- 同时读取垂直和水平滚轮增量，选择绝对值较大的轴，兼容不同蓝牙鼠标。
- 图片处于最小缩放比例时：
  - 向上滚动：上一张；
  - 向下滚动：下一张。
- 图片已放大时保留滚轮缩放行为。
- 增加约 150～200ms 防抖，避免高分辨率滚轮一次触发多次翻页。
- 消费已经处理的滚轮事件，避免 Pager 或父容器重复响应。

### 关键日志

- 滚轮方向、当前缩放比例、触发翻页或缩放结果。
- 被防抖忽略的连续事件使用 Debug 级别日志。

## 6. 问题五：超大本地仓库扫描

### 已确认瓶颈

- `DocumentFile.listFiles()` 后，对每个节点反复调用 `name/type/isFile/isDirectory`，可能触发多次 SAF Binder 查询。
- 收集图片后又逐图片调用 `ContentResolver.query()` 提取元数据，数十万张图片会产生数十万次跨进程查询。
- `findAlbumId()` 对每张图片遍历全部相册并重复 URL 解码，复杂度接近 `图片数 × 相册数`。
- 扫描过程先把所有图片记录保存在内存，超大仓库存在明显内存压力。
- 当前“暂停”实际取消任务，恢复时从头扫描，只是对显示进度做偏移，不是真实续扫。
- 遍历总量未知时仍显示模拟百分比，容易长时间停在某一进度，误导用户。

### 第一阶段：扫描核心提速

- 使用 `DocumentsContract.buildChildDocumentsUriUsingTree()` 与目录游标直接遍历。
- 每个目录只执行一次 Provider 查询，并在同一游标中读取：
  - document ID
  - display name
  - MIME type
  - size
  - last modified
- 遍历阶段直接获得图片元数据，不再为每张图片执行第二次查询。
- 预先构建规范化目录路径到相册 ID 的 Map，实现 O(1) 匹配；禁止在图片循环中反复遍历 Map 和 URL 解码。
- 使用 500～1000 条批次流式写入临时扫描结果，避免全部图片驻留内存。
- 扫描开始后继续保留旧索引；只有遍历和临时索引全部成功后，才在事务中切换新索引。
- 取消、权限失效或 Provider 异常时删除临时结果，旧索引继续可用。

### 第二阶段：真实长任务与恢复

- 使用前台 `WorkManager` 执行扫描任务，使其不依赖 Repository 页面或 ViewModel 生命周期。
- 扫描会话持久化以下状态：
  - 仓库 ID、开始时间和当前阶段；
  - 待遍历目录队列与已完成目录；
  - 已发现图片数、已索引图片数、失败目录数；
  - 暂停、取消状态。
- 暂停只在目录或数据库批次边界停下并保存检查点；恢复后继续剩余目录，不从头开始。
- 应用进程被杀或设备重启后，WorkManager 根据持久化会话继续任务。
- 同一仓库只允许一个唯一工作；多个仓库的并发上限保持较低，避免 SAF Provider 和存储设备争用。

### UI 方案

仓库卡片分阶段显示：

- `遍历目录`：不确定进度条，显示已检查目录与已发现图片数量。
- `建立索引`：确定进度条，显示已索引 / 已发现数量。
- `生成封面`：显示已处理相册数量。

同时显示：

- 实时速度（张/秒）；
- 已耗时；
- 在总量已知后显示预计剩余时间；
- 暂停、继续、取消操作；
- “后台扫描中”状态和通知入口。

取消或失败时明确提示“旧索引仍保留”，不显示虚假的 100% 或进度偏移。

### 性能日志与验收指标

为每个扫描会话生成稳定 ID，并记录：

- Provider 类型与根 URI（日志中避免泄露不必要的完整私人路径）；
- 目录查询次数、发现目录数和图片数；
- 遍历、索引、封面三个阶段耗时；
- 每批写入数量与耗时；
- 平均/最近窗口扫描速度；
- 暂停、恢复、取消和进程恢复原因；
- 无权访问或查询失败的目录数量与示例。

真机至少执行三类验证：

1. 小仓库功能回归，结果数量与旧扫描一致。
2. 中型仓库优化前后对比，采集总耗时、Binder 查询数量趋势、CPU 与内存快照。
3. 长任务暂停、恢复、切后台和进程重启验证，确认不会从头开始且旧索引始终可用。

不在缺少真实大仓库样本时承诺固定倍数；以真机日志和同一仓库前后数据作为结论。

## 10. 实施结果（2026-06-22）

### 已完成

- 数据库导出使用 `VACUUM INTO`，备份包含 manifest；导入兼容新旧 ZIP 和原始 DB，并执行路径、完整性、版本及回滚校验。
- 远程服务持有 PARTIAL_WAKE_LOCK/WifiLock/MulticastLock，并在前后台、屏幕唤醒和网络/IP 变化时自愈。
- 相册协议增加可选 `coverImageId`；服务端生成下采样缩略图，客户端使用远程缩略图 Loader。
- 鼠标滚轮独立处理，Pager 单一执行翻页并限制高速连续事件。
- SAF 遍历阶段每目录单次 Provider 查询；图片元数据写入磁盘 Spool，不再让全部图片对象驻留 JVM 堆。
- 新索引切换由 Room 事务保护，失败或取消时保留旧索引。
- 前台 WorkManager 支持息屏/切后台扫描，UI 分阶段显示已发现、已扫描、总数和剩余数。
- Android 15 扫描启动闪退已通过显式 `dataSync` FGS 类型修复。

### 验证证据

- `:app:assembleDebug :app:testDebugUnitTest` 成功；当前单元测试任务为 `NO-SOURCE`。
- V2254A 冷启动正常，crash buffer 无新增异常。
- 后台 15 秒后 HTTP `/api` 可访问；远程服务保持 foreground，类型为 `dataSync`。
- 5224 个相册中 4425 个返回封面 ID；抽样缩略图响应为 JPEG 109298 bytes。
- 修复扫描 FGS 类型后 Worker 通知 ID 3002、类型 `0x1` 正常运行，观察 20 秒进程存活且无新增 crash。

### 未完成

- 目录队列与完成目录的持久化检查点；暂停/进程重建后目前重新遍历。
- 大仓库完整扫描耗时、峰值内存、CPU/Binder 趋势及优化前后对比。
- 多台设备间长时间后台、Wi-Fi 切换和不同蓝牙鼠标的矩阵测试。
