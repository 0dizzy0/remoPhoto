# remoPhoto 测试用例

更新时间：2026-06-22

本文整理 Phase 4 远程仓库、设置导航、分类、数据库迁移及大型仓库真机验证过程，作为后续单元测试、Compose UI 测试、仪器测试和 adb 端到端自动化的基准。

## 1. 测试目标与分层

| 层级 | 建议工具 | 主要覆盖范围 |
|------|----------|--------------|
| JVM 单元测试 | JUnit、Coroutines Test | 页数计算、名称定位、排序、缓存过期策略、DTO 兼容解析 |
| 数据库仪器测试 | AndroidX Test、Room MigrationTestHelper | v3→v4 迁移、批量同步、标签关联、事务回滚 |
| Compose UI 测试 | Compose UI Test | 导航、设置项、分页、查找、刷新状态、空状态 |
| HTTP 契约测试 | MockWebServer 或本机 HTTP Server | 新旧 `/api/albums`、分页图片接口、错误与超时 |
| 真机端到端 | adb + UIAutomator/Compose semantics | 安装升级、网络切换、缓存命中、性能、崩溃和日志 |

自动化脚本不得依赖本文记录的绝对坐标。应为关键节点增加稳定的 `testTag`，或通过 `contentDescription`、文本和 UI 树边界动态定位。

## 2. 基准环境与测试数据

### 2.1 已验证环境

| 项目 | 值 |
|------|----|
| 客户端设备 | V1981A，adb serial `1e6cd7cc` |
| 大型远端设备 | V2254A，HTTP `192.168.10.7:8080`（测试时地址） |
| 应用 | `com.remophoto`，Debug，targetSdk 35 |
| 数据库 | Room v4 |
| 大型远程仓库 | 5225 个相册、267995 张图片 |
| 每页数量 | 20 |
| 预期总页数 | 262 |

IP、serial、仓库名称和数据量必须参数化，不得写死到自动化脚本。

### 2.2 建议固定数据集

准备以下可重复构建的数据夹具：

1. 空仓库：0 个相册。
2. 小仓库：4 个相册，其中 2 个有封面。
3. 分页边界仓库：19、20、21、40、41 个相册各一份。
4. 大仓库：至少 5000 个相册；推荐复用 5225/267995 样本。
5. 修改时间仓库：至少 4 个相册，时间戳顺序明确且包含一个 `lastModified=0` 的兼容样本。
6. 分类仓库：一个远程相册关联临时分类 `QA-Remote`。
7. 协议样本：新版响应含 `coverImageId`、`lastModified`；旧版响应缺少两个可选字段。

## 3. 通用 adb 验证模板

```powershell
$serial = '<adb-serial>'
$apk = 'app\build\outputs\apk\debug\app-debug.apk'

adb -s $serial install -r $apk
adb -s $serial logcat -c
adb -s $serial shell am force-stop com.remophoto
adb -s $serial shell am start -W -n com.remophoto/.MainActivity
adb -s $serial shell uiautomator dump /sdcard/remophoto-ui.xml
adb -s $serial logcat -b crash -d
```

每次端到端测试至少保存：构建版本、设备型号/API、开始结束时间、关键 AppLogger 行、crash buffer、必要的 `dumpsys meminfo`/`gfxinfo`。

## 4. 构建、安装与数据库

### TC-ENV-001 构建与单元测试

- 前置：JDK 17、Android SDK 可用。
- 步骤：运行 `./gradlew :app:testDebugUnitTest :app:assembleDebug --console=plain`。
- 断言：任务成功；APK 存在；分页测试中 `5225 / 20 = 262`。

### TC-DB-001 v3→v4 覆盖安装迁移

- 前置：安装含 Room v3 数据的旧 APK，数据库中存在本地/远程仓库、分类和关联。
- 步骤：使用 `adb install -r` 覆盖安装 v4；冷启动。
- 断言：无 Room migration 异常；仓库、相册、分类和关联数量不减少；`albums.last_modified` 存在，并由已有图片最大修改时间回填。
- 自动化建议：使用 `MigrationTestHelper` 校验 schema 和回填 SQL；真机只做升级烟雾测试。

### TC-DB-002 远程同步事务和主键稳定性

- 前置：远程相册 A 已缓存并关联分类。
- 步骤：执行两次相册元数据同步。
- 断言：A 的本地主键不变；分类关联仍存在；同步不删除已缓存图片索引；服务端删除的相册及其图片索引被删除。

## 5. 设置与导航

### TC-NAV-001 设置切回相册列表

- 步骤：主界面进入设置；通过底部“相册列表”Tab 返回；重复 10 次。
- 断言：每次都返回仓库列表或当前相册状态；无空白页、无导航失败、无 `AndroidRuntime` 异常。

### TC-SET-001 设置页统计和缓存栏目

- 步骤：进入设置并滚动到“存储空间”“远程缓存”。
- 断言：已索引图片、图片文件总大小、缩略图缓存和原图缓存均显示有效值，不长期停留在“计算中…”；读取失败时显示“读取失败”。
- 已验证样本：724 张、177.9 MB、远程缓存 0 B/0 B。

### TC-SET-002 清除远程缓存

- 前置：缩略图和原图缓存均非空。
- 步骤：点击清除远程缓存。
- 断言：Coil 内存和磁盘缓存均清空；统计刷新为 0 B；日志包含清理结果；相册列表不崩溃。

## 6. 相册排序与查找

### TC-ALB-001 相册排序与图片排序相互独立

- 步骤：分别修改“图片默认排序”和“相册列表排序”。
- 断言：修改相册列表排序不改变相册内图片顺序，反之亦然；重启后两项分别持久化。

### TC-ALB-002 相册修改时间排序

- 前置：4 个相册的 `last_modified` 已知且不同。
- 步骤：选择“修改时间（旧→新）”，记录顺序；再选择“修改时间（新→旧）”。
- 断言：两次顺序互为反向；同时间戳按名称稳定排序；旧服务端缺字段时解析为 0 且不崩溃。
- 注意：远程设备必须升级至支持 `/api/albums.lastModified` 的版本，才能验证真实远程时间顺序。

### TC-ALB-003 名称模糊定位

- 前置：目标相册位于非首页，例如名称包含 `Fanbox`。
- 步骤：输入大小写不同的名称片段并查找。
- 断言：切换到首个命中相册所在页；列表总数不变；不进入筛选模式；无匹配时显示提示。

## 7. 分页

### TC-PAG-001 响应式显示分页栏

- 前置：5225 个相册，每页 20。
- 步骤：从主界面进入远程仓库，等待数据库缓存完成展示。
- 断言：显示 `< 1 2 … 262 >`；分页栏不得因初始总页数为 1 而永久隐藏。

### TC-PAG-002 首页、中间页和末页切换

- 步骤：依次进入第 1、2、262 页。
- 断言：当前页状态正确；第 2 页首项不同于第 1 页；末页数据不重复越界；上一页/下一页启用状态正确。
- 已验证：第 2 页首项发生变化，第 262 页可正常加载。

### TC-PAG-003 分页边界参数化

| 相册数 | 每页 | 预期页数 |
|-------:|-----:|---------:|
| 0 | 20 | 1 |
| 19 | 20 | 1 |
| 20 | 20 | 1 |
| 21 | 20 | 2 |
| 40 | 20 | 2 |
| 41 | 20 | 3 |
| 5225 | 20 | 262 |

## 8. 远程数据库缓存与刷新

### TC-CACHE-001 首次同步后写入数据库

- 前置：远程仓库尚无本地相册缓存。
- 步骤：首次进入仓库并等待同步结束；退出后断网再次进入。
- 断言：首次同步写入完整相册元数据；断网时仍可显示数据库中的相册列表和封面缓存占位状态。

### TC-CACHE-002 30 分钟内缓存命中

- 前置：`last_scan_time` 距当前少于 30 分钟，缓存中有 5225 个相册。
- 步骤：记录“远程相册同步完成”日志计数；退出并重新进入仓库；等待 5 秒。
- 断言：相册立即显示；同步日志计数不增加；日志包含“远程相册缓存仍有效，跳过同步”。
- 已验证：缓存树构建约 98～106 ms，重新进入未触发网络同步。

### TC-CACHE-003 缓存过期后台刷新

- 前置：缓存存在，但 `last_scan_time` 超过 30 分钟。
- 步骤：进入仓库。
- 断言：先显示旧缓存和分页，再显示刷新进度；刷新期间列表可操作；成功后原位更新；失败时旧缓存仍保留。

### TC-CACHE-004 手动刷新

- 步骤：点击“刷新远程相册缓存”。
- 断言：触发一次强制同步；按钮显示进行中状态并禁止重复点击；列表和分页保持可见；结束后状态恢复。
- 已验证：5225 个相册完整刷新约 3.7～4.7 秒。

## 9. 分类和离线行为

### TC-CAT-001 远程标签跨同步保留

- 前置：创建临时分类 `QA-Remote`。
- 步骤：长按远程相册并加入分类；退出仓库；重新进入触发同步；打开该分类。
- 断言：远程相册仍在分类中；相册本地主键和交叉表记录未改变。
- 清理：删除 `QA-Remote` 分类。

### TC-CAT-002 远程离线时分类隐藏

- 前置：分类中只有远程相册。
- 步骤：关闭 Wi-Fi；重新进入分类；记录 UI；恢复 Wi-Fi并重新同步。
- 断言：离线时显示“暂无相册”，不显示远程缓存相册；恢复后远程仓库状态回到“已连接”。
- 清理要求：测试脚本必须使用 `try/finally` 恢复 Wi-Fi。

## 10. HTTP 协议兼容

### TC-API-001 新版 `/api/albums`

- 步骤：请求 `/api/albums`。
- 断言：响应包含 `albums`；每项包含基本字段及可选 `coverImageId`、`lastModified`；远程中继仓库不被二次共享。
- 已验证样本：本机 v4 API 返回 3 个本地相册，含 `lastModified`，请求约 425 ms。

### TC-API-002 旧服务端兼容

- 前置：MockWebServer 返回不含 `coverImageId`、`lastModified` 的相册 JSON。
- 断言：客户端解析成功；两个字段分别为 `null` 和 `0`；列表、分页和名称排序可用。

### TC-API-003 网络错误保留缓存

- 前置：数据库已有缓存。
- 步骤：让 `/api/albums` 返回超时、连接拒绝和 HTTP 500。
- 断言：连接标记为 `ERROR`；缓存列表不被删除；重试次数符合配置；日志包含最终失败原因。

## 11. 大型仓库性能与稳定性

### TC-PERF-001 大型相册树构建

- 数据：5225 个无父级相册。
- 断言：构建复杂度为 O(n)；Debug 真机目标小于 500 ms；日志只输出汇总，不逐相册输出封面路径。
- 已验证：约 98～106 ms。

### TC-PERF-002 大型同步批量事务

- 步骤：对已缓存的 5225 个相册执行手动刷新。
- 断言：使用批量 `updateAll/insertAll`；事务失败时数据库保持旧状态；标签和图片缓存不丢失。
- 已验证：完整刷新约 3.7～4.7 秒。

### TC-PERF-003 内存与崩溃

- 步骤：进入大型仓库，切换到第 262 页，执行 `dumpsys meminfo` 和 crash buffer 检查。
- 断言：无 OOM、ANR、`FATAL EXCEPTION`；记录 Java Heap、PSS、RSS，不将单次 Debug 数值作为发布硬门槛。
- 已验证快照：Java Heap 约 38 MB、PSS 约 226 MB、RSS 约 345 MB。

### TC-LOG-001 日志可观测性

- 断言：关键日志至少包含仓库 ID/名称、相册数、缓存年龄、同步新增/复用/删除数量、阶段耗时；正常协程取消不得记录为错误；大型仓库单次进入不得产生数千条逐相册日志。

## 12. 自动化落地顺序

1. 将 `AlbumListLocatorTest` 扩展为全部分页边界参数化测试。
2. 增加 `AlbumSortOrder` 和稳定排序单元测试。
3. 使用 Room `MigrationTestHelper` 覆盖 TC-DB-001/002。
4. 使用 MockWebServer 覆盖 TC-API-001～003 和缓存 TTL。
5. 为仓库卡片、分页栏、查找框、刷新按钮、分类空状态增加稳定 `testTag`。
6. 编写 Compose UI 测试覆盖 TC-NAV、TC-SET、TC-ALB、TC-PAG。
7. 将 adb 网络切换、meminfo、crash buffer 和日志断言放入独立真机回归脚本，不在普通 JVM CI 中执行。
