## Alpha 后第一轮迭代

目标：把首个 alpha 反馈转化为可回归的质量体系。

| 优先级 | 任务 | 价值 |
| --- | --- | --- |
| P0 | 修复安装、启动、迁移、远程连接类崩溃 | 保证 alpha 可继续试用 |
| P0 | 补分页、排序、名称定位 JVM 测试 | 一条 Gradle 命令固定最常用相册行为，无设备和大型仓库依赖 |
| P0 | 补导入 manifest/version、白名单和路径安全 JVM 测试 | 低成本保护备份输入边界 |
| P1 | 补远程 API DTO、分页参数和错误映射 JVM 测试 | 不连接真实 Wi-Fi 即可固定设备互联契约 |
| P1 | 补最小 Room v3→v4 migration 测试 | 先保护最近升级路径，避免一次性维护全历史 fixture |
| P1 | 按已发现缺陷增加备份回滚或缓存集成测试 | 用实际风险决定投入 |
| P2 | 增加 1～3 条基础 Compose 核心旅程 | 覆盖 JVM 无法验证的导航和持久化 |
| P2 | 大仓库性能、完整历史迁移和 adb 自动回归 | 有稳定数据、设备或重复执行收益时再投入 |

## Beta 候选范围

Beta 阶段应解决 alpha 暴露出的稳定性问题，并补齐自动化回归。可考虑纳入：

- SMB 电脑共享仓库。
- HTTPS/TLS 或局域网访问令牌。
- 扫描任务目录队列级持久化检查点。
- 更完整的 UI 自动化测试。
- 远程服务安全和隐私说明。
- README 的真实截图、安装包链接和更完整安装说明。

以下事项不建议在 alpha 前阻塞：

- 视频播放。
- 多用户权限管理。
- Play Store 上架材料。
- 固定性能指标承诺。

## 技术债清单

| 项目 | 当前影响 | 建议处理时间 |
| --- | --- | --- |
| 自动化测试覆盖低 | 回归主要依赖手工 | alpha 后第一轮 |
| 明文 HTTP 无认证加密 | 可信局域网外不安全 | Alpha 保留边界说明，Beta 评估认证与 TLS |
| WorkManager 扫描恢复不精确 | 进程被杀后可能重新遍历 | beta |
| 文档历史堆叠 | 维护成本高 | 本轮已收敛 |

## 发布命令

基础门禁：

```powershell
.\gradlew.bat :app:testDebugUnitTest :app:lintDebug :app:assembleRelease --console=plain
```

发布前建议：

```powershell
git status --short
Get-FileHash app\build\outputs\apk\release\<signed-apk-name>.apk -Algorithm SHA256
```

## 维护规则

1. 新任务进入本文件时必须带优先级和验收方式。
2. 已完成的大段过程不继续保留在本文件，只在[项目状态](项目状态.md)记录摘要。
3. 问题根因和修复细节写入[问题日志](06_问题日志.md)。
4. 测试设计写入[核心功能测试用例](../testing/07_测试用例.md)，单次结果使用[测试执行记录模板](../testing/测试执行记录模板.md)。
5. 发布门禁写入[首个 Alpha Release 计划](../releases/08_首个AlphaRelease计划.md)。
