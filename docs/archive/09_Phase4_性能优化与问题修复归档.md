# Phase 4 性能优化与问题修复归档

更新时间：2026-06-27

## 1. 文档定位

本文是 Phase 4 远程仓库和大仓库性能修复的归档摘要。当前开发计划以[开发计划](../project/05_开发计划.md)为准，Alpha 发布以[首个 Alpha Release 计划](../releases/08_首个AlphaRelease计划.md)为准。

## 2. 背景

Phase 4 引入 HTTP/mDNS 远程仓库后，真机反馈集中在五类问题：

1. 数据库导入导出一致性和版本兼容。
2. 息屏、后台、网络变化后远程服务不可用。
3. 远程相册封面加载慢或不显示。
4. 蓝牙鼠标滚轮在全屏浏览中不可用或抖动。
5. 本地大仓库扫描慢、内存压力高、长任务 UI 不清晰。

## 3. 已完成修复

### 数据库导入导出

- 使用 SQLite `VACUUM INTO` 生成一致性快照。
- 导出 manifest，记录格式版本、数据库版本、创建时间和远程连接统计。
- 导入前执行白名单解压、路径穿越防护、`integrity_check` 和版本检查。
- 替换前创建回滚快照。
- 远程连接元信息保留，凭据不导出，导入后状态重置为断开。

### 远程服务自愈

- HTTP 服务改为前台服务。
- 持有必要的 Wi-Fi lock、multicast lock 和 wake lock。
- App 启动、回前台、网络恢复、屏幕唤醒和 IP 变化时触发自愈检查。
- 服务启动失败时保留已有引用，避免误清理 mDNS。

### 远程相册封面

- `/api/albums` 增加 `coverImageId`。
- `/api/image/{id}/thumb` 返回下采样缩略图，避免把原图作为封面传输。
- 客户端使用远程 thumbnail loader 加载封面。
- 兼容旧服务端缺少 `coverImageId` 的情况。

### 鼠标滚轮翻页

- 全屏浏览增加 pointer scroll 处理。
- 非缩放状态滚轮翻页。
- 缩放状态滚轮优先缩放。
- 增加节流，避免高速滚轮和 pager 状态竞争。

### 大仓库扫描

- SAF 扫描改为每目录单次游标查询。
- 结果写入磁盘 spool，避免大量对象常驻内存。
- 批量写入临时索引。
- 旧索引删除、新相册和新图片写入放入同一 Room 事务。
- 引入前台 WorkManager 继续执行长任务。
- UI 展示阶段化数量：已检查目录、已发现图片、已扫描/总数/剩余。

### 大型远程仓库

- 远程相册列表先读 Room 缓存。
- 缓存 30 分钟内有效。
- 过期时先展示缓存，再后台刷新。
- 同步改为批量事务，复用稳定远程路径对应的本地主键。
- 相册树构建从 O(n²) 优化为按 parentId 分组的 O(n)。
- 分页总页数改为响应式状态。

## 4. 验证摘要

已进行过的验证包括：

- 真机启动远程服务，无新增 crash。
- HTTP `/api` 和 `/api/albums` 可访问。
- mDNS 注册和发现可用，能过滤自身。
- 远程仓库可添加、同步和分页。
- 大型远程仓库样本约 5225 个相册、267995 张图片，分页可跳转到第 262 页。
- 远程分类重同步后关联保留。
- 远程离线时分类隐藏远程相册，恢复网络后可重新连接。

这些是手工验证结果，后续仍需自动化测试固化。

## 5. 残留限制

| 限制 | 说明 | 后续计划 |
| --- | --- | --- |
| 目录队列级检查点未实现 | 进程被杀后扫描可能重新遍历 | Beta 评估 |
| 未完成完整前后性能对比 | 只有单次真机快照，不能承诺固定加速倍数 | 建立性能测试脚本 |
| 厂商后台限制不可控 | 部分设备可能杀前台/后台任务 | 通过日志收集反馈 |
| HTTP 明文 | 只适合可信局域网 | 后续考虑令牌或 HTTPS |
| 自动化测试不足 | 目前主要是手工验证 | alpha 后补 migration/API/UI 测试 |

## 6. 关键日志要求

这些流程后续仍需保留摘要日志：

- 数据库导入导出：版本、manifest、校验、替换、回滚。
- 服务自愈：原因、端口、设备名、IP、注册结果。
- 远程同步：连接、缓存命中、刷新耗时、新增/复用/删除数量。
- 大仓库扫描：目录数、图片数、spool 文件、批量写入耗时、取消或失败原因。
- 分页和树构建：相册数量、耗时、页数。

## 7. 后续建议

1. 为 Room migration 增加自动化测试。
2. 为远程 API 增加新旧协议兼容测试。
3. 为导入导出增加格式和回滚测试。
4. 将大仓库性能验证脚本化。
5. 把 release 日志等级和脱敏策略落地。
