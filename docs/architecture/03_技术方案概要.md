# remoPhoto 技术方案概要

更新时间：2026-06-27

## 1. 技术栈

| 层级 | 方案 |
| --- | --- |
| 语言 | Kotlin |
| UI | Jetpack Compose + Material 3 |
| 架构 | MVVM + Repository + UseCase |
| 异步 | Kotlin Coroutines + Flow/StateFlow |
| 数据库 | Room，当前 schema version 4 |
| 设置 | DataStore Preferences |
| 图片加载 | Coil + GIF/WebP 支持 |
| 本地文件访问 | SAF / DocumentFile |
| 后台任务 | WorkManager 前台任务 |
| 远程服务 | NanoHTTPd |
| 服务发现 | JmDNS / mDNS |
| 日志 | `AppLogger` 文件 + Logcat |

## 3. 架构边界

### UI 层

- Compose Screen 只负责展示状态和发出用户意图。
- ViewModel 持有 `StateFlow` 或 Compose state。
- 长任务状态通过 repository/usecase 回流到 UI。
- 大型列表避免在组合阶段直接执行重计算。

### Domain 层

- UseCase 负责扫描、同步、排序、分类、封面等业务动作。
- 计算密集逻辑放到 `Dispatchers.Default`。
- IO 和网络逻辑放到 `Dispatchers.IO`。

### Data 层

- Room 是本地索引和缓存的权威数据源。
- SAF 目录授权是本地仓库访问前提。
- 远程仓库元数据同步到本地数据库，不直接把 UI 绑定到网络响应。
- Keystore 只保存凭据，导入导出不携带凭据。

## 4. 数据库

当前 Room version：4。

核心表：

- `image_repositories`：本地/远程仓库。
- `remote_connections`：远程连接元信息。
- `albums`：相册和层级关系，含 `last_modified`。
- `images`：图片索引。
- `categories`：分类。
- `album_category_cross_ref`：相册和分类多对多关系。

迁移链：

- v1 -> v2：增加查询索引。
- v2 -> v3：增加远程连接表和仓库关联。
- v3 -> v4：增加相册 `last_modified` 并按图片最大修改时间回填。

待改进：

- 配置 Room `schemaLocation`。
- 提交 schema JSON。
- 增加 migration 自动化测试。

## 5. 本地扫描方案

目标：支持数十万图片级别仓库，不阻塞 UI，不破坏旧索引。

当前策略：

- 使用 SAF 持久化授权读取目录。
- 每个目录尽量使用单次游标查询，减少逐图片 Provider 查询。
- 扫描结果写入磁盘 spool，避免大量对象常驻内存。
- 批量写入临时索引。
- 完整成功后在 Room 事务内切换新索引。
- 取消、失败或权限异常时保留旧索引。
- 使用前台 WorkManager 让长任务在后台/息屏时继续执行。

残留限制：

- 进程被系统杀死后，目录队列级检查点尚未实现，可能需要重新遍历。

## 6. 图片加载方案

本地图片：

- 使用 Coil 加载 SAF/本地 URI。
- 缩略图和全屏图使用不同尺寸策略。
- GIF/WebP 使用对应 decoder。

远程图片：

- `RemoteImageFetcher` 支持 HTTP 图片和缩略图 URL。
- 远程缩略图和原图使用独立磁盘缓存目录。
- 相册封面使用远程 thumbnail URL，避免直接传输大图。

## 7. 远程仓库方案

当前优先支持 Android 设备之间的局域网共享：

- 服务端：NanoHTTPd 暴露本地仓库 API。
- 发现：JmDNS 注册 `_remophoto._tcp`。
- 客户端：mDNS 发现或手动 IP 添加。
- 同步：先拉取相册元数据，再按需拉取图片元数据。
- 缓存：相册和图片元数据写入 Room。
- 离线：连接失败时保留已有缓存，状态标记为错误或断开。

安全边界：

- 当前是明文 HTTP，仅面向可信局域网。
- 不支持公网访问。
- 不支持 HTTPS/TLS。
- 不导出远程凭据。

详见[远程仓库协议决策](04_远程仓库协议选型建议.md)。

## 8. 导入导出方案

导出：

- 使用 SQLite `VACUUM INTO` 创建一致性数据库快照。
- 写入 manifest，包含格式版本、数据库版本、创建时间和远程连接统计。
- 导出 DataStore 设置。
- 不导出 Coil 缓存和凭据。

导入：

- 先解压到缓存临时目录。
- 执行路径穿越防护和文件白名单校验。
- 执行 `integrity_check` 和版本检查。
- 远程连接状态重置为 `DISCONNECTED`。
- 替换前创建回滚快照。
- 替换失败时恢复原数据库和设置。

## 9. UI 与性能策略

- 大列表使用分页，避免一次渲染过多节点。
- 相册树构建按 parentId 分组，避免 O(n²)。
- 相册列表总页数使用响应式状态。
- 远程仓库优先展示 Room 缓存，缓存过期时后台刷新。
- 设置页缓存统计在 IO 线程执行。
- 高频滚动状态避免直接在组合中读取。

## 10. 日志策略

关键流程必须记录：

- 扫描：repoId、阶段、目录数、图片数、耗时、失败目录。
- 远程服务：启动、停止、自愈、IP、mDNS 注册。
- 远程同步：连接、缓存命中、刷新、新增/复用/删除数量、耗时。
- 导入导出：manifest、版本、校验、替换、回滚。
- 性能：大仓库树构建、分页、缓存统计耗时。

Alpha 前建议：

- Debug 保留 DEBUG。
- Release 默认 INFO 或 WARN。
- 大仓库只记录汇总，不逐相册刷日志。
- 对完整路径和局域网地址做必要脱敏或减少输出。

## 11. 依赖与镜像

项目使用 Gradle Wrapper。国内网络较慢时优先使用已配置的阿里云 Maven 镜像，并保留 Google/MavenCentral 作为兜底。

新增依赖前需要确认：

1. 目标版本在镜像仓库或官方仓库可达。
2. 依赖确实解决核心问题，避免为小功能引入重库。
3. Android minSdk 29 和 targetSdk 35 下行为明确。
