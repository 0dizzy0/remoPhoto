## 文档约定

`docs/` 是项目文档的统一入口。文档按职责分目录维护，避免需求、实现、计划和测试结论混写。

- 产品文档说明“为什么做、做什么”。
- 架构文档说明“当前如何实现、边界是什么”。
- 项目文档维护“当前状态、下一步和问题追踪”。
- 测试文档定义“如何验证、需要留下什么证据”。
- 发布文档定义“何时可以发版、发版时检查什么”。
- 归档文档只用于追溯，不作为当前工作的事实来源。

所有文本文件使用 UTF-8 编码。文档中的相对链接必须可从 GitHub 直接访问。

## 更新规则

1. 新需求先更新需求分析，再同步功能规格和测试用例。
2. 用户可见行为变化必须同时更新对应测试用例。
3. 实现细节只进入架构文档，不复制到需求文档。
4. 已完成阶段在项目状态中保留摘要，详细过程移入归档。
5. 问题日志只保留“现象、影响、根因、修复、验证、残留风险”。
6. 发布计划描述策略，发布检查清单记录一次具体执行，两者不要混写。
7. 扫描、远程同步、导入导出和服务自愈等关键路径必须在测试断言中包含可观测日志要求。
8. 新增文档后更新本索引，并检查 GitHub 相对链接。
9. AI 代理、Skills、MCP 或本地工具链发生变化时，更新 AI 辅助开发说明；区分“实际使用”“已安装”和“项目级配置”。
