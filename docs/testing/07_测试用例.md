# remoPhoto 核心功能测试用例

更新时间：2026-07-07

## 1. 文档目标

本文定义 remoPhoto 核心功能的长期回归基线，供 Alpha 发布验证和后续开发使用。设计目标是让个人开发者能在普通开发机和一台 Android 设备上完成最低门禁，再按改动范围追加专项测试。测试范围来自：

- [需求分析](../product/01_需求分析文档.md)
- [功能规格](../product/02_功能规格说明.md)
- [技术方案](../architecture/03_技术方案概要.md)
- [远程仓库协议决策](../architecture/04_远程仓库协议选型建议.md)
- [首个 Alpha Release 计划](../releases/08_首个AlphaRelease计划.md)
- 当前应用路由、设置项、HTTP API 和已有 JVM 测试

本文是“测试设计”，不保存某次执行结果。每轮执行请复制[测试执行记录模板](测试执行记录模板.md)。

## 2. 测试原则

### 2.1 优先级

| 优先级 | 定义 | 发布要求 |
| --- | --- | --- |
| P0 | 安装、启动、本地仓库扫描和图片浏览等最小核心闭环 | 每个候选包执行最低 Smoke；模块变更时补齐相关 P0 |
| P1 | 远程仓库、导入导出、迁移、异常恢复等模块能力 | 首次 Alpha 至少执行一次；后续在模块变更时执行 |
| P2 | 大数据量、长稳、完整兼容矩阵和体验边界 | 有真实风险或可用环境时执行，不阻塞个人开发者日常发布 |

优先级描述业务风险，不等于自动化实施顺序。自动化实施还要考虑收益、开发成本和运行门槛；优先选择可在 JVM 中稳定、快速、无外部环境运行的核心规则。

### 2.2 测试层级

| 层级 | 适用范围 | 当前策略 |
| --- | --- | --- |
| JVM | 纯计算、排序、分页、查找、DTO/manifest 校验 | 第一优先；复用现有 JUnit，默认不新增服务和设备依赖 |
| Android 集成 | Room 迁移、DataStore、SAF、WorkManager、缓存 | 仅覆盖 JVM 无法验证的高风险路径；可用单个模拟器或真机运行 |
| Compose UI | 导航、状态、设置、分页、空错页面 | 只保留少量核心旅程，不追求控件级全量覆盖 |
| HTTP 契约 | API 路由、字段、状态码、兼容性 | 优先测试 DTO 和纯规则；确需路由验证时使用进程内假数据 |
| 真机 E2E | SAF、前台任务、mDNS、图片加载、系统生命周期 | 保留最小 Smoke；双机远程测试作为首次 Alpha 和远程模块变更门禁 |

### 2.3 低门槛约束

新增自动化用例默认满足：

- 单条 JVM 用例不依赖网络、Wi-Fi、真实图片库、固定磁盘路径或个人数据。
- 使用代码生成的小型元数据、少量可公开测试图片或临时目录；测试结束自动清理。
- 分页边界按当前每页数量 `P` 使用 `0/1/P/P+1/2P/2P+1` 等最小数据证明规则，无须创建数千个真实相册。
- 常规随机数据使用固定种子；模糊测试可使用变化种子，但失败时必须记录种子以便复现。时间、网络和存储通过可替换依赖控制，避免等待真实 TTL。
- JVM 核心套件目标在普通开发机 5 分钟内完成；超出后拆到专项任务。
- 关键失败输出用例 ID、数据规模、预期值和实际值，不输出完整私人路径、相册名或图片内容。

### 2.4 可扩展性与断言边界

测试优先验证用户可观察行为和跨版本契约，不绑定可替换实现：

- 必须长期稳定的不变量：不崩溃、不越权、不泄漏凭据、不静默损坏数据，已发布且仍受支持的数据和协议可兼容。
- 可演进的当前基线：默认分页大小、缓存时长、图片格式、缩略图编码、导航布局和内部存储方案。断言应从当前配置、能力清单或版本化规格读取，不在多个用例中重复写死。
- 不直接断言内部类名、第三方库、数据库表实现、协程调度方式或日志原文。只有平台要求或已公开契约明确规定时才例外。
- UI 用例验证页面可达、状态变化和用户结果，不依赖固定坐标、完整控件树或易变化的展示文案。
- 新增图片格式、仓库类型、排序方式或协议版本时，优先向参数化能力矩阵追加数据；不复制整套结构相同的用例。
- 有意修改产品行为时，应在同一改动中更新功能规格、版本化契约和测试期望。测试用于暴露未声明变化，不用于永久冻结合理设计。

精确数值只在安全上限、数据格式版本、数据库迁移或已发布 API 契约中作为硬约束；普通默认值使用符号参数表示。例如当前每页数量记为 `P`，缓存有效期记为 `T`。

参数来源遵循以下顺序：

| 参数类型 | 测试取值来源 | 约束 |
| --- | --- | --- |
| 用户可配置值，如 `P`、播放间隔 | 被测版本公开配置或共享测试 fixture | 用独立公式验证结果，不能直接复用被测计算函数作为期望值 |
| 已发布协议限制，如最大页大小 `M` | 对应契约基线或 API/格式版本的 fixture | 与生产实现分离，协议升级时新增版本，不覆盖旧版本 |
| 能力集合，如图片格式、排序方式 | 功能规格对应的参数化 fixture | 新增能力只增加参数和该能力特有断言 |
| 安全与数据完整性规则 | 独立固定的恶意/损坏输入 fixture | 不随普通配置放宽 |

### 2.5 结果定义

- `PASS`：全部断言满足，Crash buffer 无新增崩溃。
- `FAIL`：至少一个断言不满足。
- `BLOCKED`：测试环境或前置条件无法满足，必须记录原因。
- `N/A`：当前测试范围明确不适用，必须记录裁剪依据。

失败用例必须保留最小复现步骤、版本、环境、脱敏日志和关联缺陷 ID。

## 3. 测试环境

### 3.1 设备矩阵

最低可运行配置：

| 角色 | 要求 | 用途 |
| --- | --- | --- |
| 主测设备 | 一台 Android 10 及以上真机；优先使用日常开发设备 | 安装、本地仓库、扫描、图片浏览和设置 |
| 远程组合 | 两台同一可信 Wi-Fi 的 Android 设备；只在远程专项使用 | mDNS、HTTP、离线恢复 |
| 可选兼容设备 | 与主测设备 Android 大版本不同的真机或模拟器 | 发布前抽查兼容性，不作为日常门禁 |

只有一台设备时，先完成 P0 本地流程；远程用例记录为本轮未执行，不应伪装成失败或阻塞本地功能开发。首次公开 Alpha 若包含远程仓库能力，仍建议在发布前完成一次双机 Smoke。

### 3.2 标准数据集

| 数据集 | 内容 | 用途 |
| --- | --- | --- |
| D0 空状态 | 无仓库、无分类 | 首次启动和空页面 |
| D1 核心小仓库 | 建议 5～10 个相册、20～100 张非隐私测试图片；至少覆盖嵌套目录、代表性静态图片、特殊字符文件名、损坏文件和非图片文件 | 本地核心 Smoke |
| D2 分页边界 | 由测试代码按当前每页数量 `P` 生成 0、1、`P`、`P+1`、`2P`、`2P+1` 条相册元数据，不要求真实图片文件 | 分页规则 |
| D3 排序与查找 | 由测试代码生成大小写、数字、同名、同图片数、同修改时间、中文和 Emoji 名称 | 稳定排序和模糊定位 |
| D4 中型可选仓库 | 可按手头数据准备 50～200 个相册、500～2000 张公开或自有测试图片 | 扫描体验抽查，不作为发布硬门禁 |
| D5 远程小仓库 | 建议 3～10 个相册、20～100 张图片；分页专项由服务端 fixture 按 `P+1` 生成元数据 | 远程核心闭环和跨页验证 |
| D6 备份样本 | 当前有效包、旧格式包、损坏 ZIP、非法条目、过高版本 manifest、损坏数据库 | 导入校验和回滚 |
| D7 大仓库专项 | 仅在有现成样本或可重复生成脚本时使用；规模按专项目标记录 | 性能、内存和长稳验证 |

测试数据不得包含未经授权的私人图片。大型真实仓库不是日常自动化、Alpha P0 或贡献代码的前置条件。D7 结果只能说明对应设备与样本表现，不能替代 D1～D3 的确定性正确性测试。

## 4. 发布前基础门禁

日常开发和提交前优先运行无设备、无 Release 签名要求的快速门禁：

```powershell
.\gradlew.bat :app:testDebugUnitTest :app:lintDebug --console=plain
```

生成发布候选包时再运行完整门禁：

```powershell
.\gradlew.bat :app:testDebugUnitTest :app:lintDebug :app:assembleRelease --console=plain
```

发布 APK 生成后，可用一键 Smoke 脚本固化安装、签名、版本、冷启动、UI dump、截图、Crash buffer、进程日志、应用文件日志、基础 Release 日志隐私扫描和内存快照等证据：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\run-smoke.ps1 `
  -Serial "<adb-serial>" `
  -ApkPath "app\build\outputs\apk\release\app-release.apk" `
  -ExpectedVersionName "<versionName>" `
  -ExpectedVersionCode <versionCode>
```

如需在同一入口中先执行 JVM、Lint 和 Release 构建，可增加 `-BuildRelease`。证据默认输出到 `.cache/qa/smoke/<timestamp>/`，该目录不进入版本控制。

| ID | 优先级 | 步骤 | 预期结果 |
| --- | --- | --- | --- |
| TC-BLD-001 | P1 | 执行 `:app:assembleDebug` | 构建成功，生成 Debug APK |
| TC-BLD-002 | P0 | 执行 `:app:testDebugUnitTest` | 全部 JVM 测试通过，报告中无跳过的 P0 测试 |
| TC-BLD-003 | P0 | 执行 `:app:lintDebug` | `BUILD SUCCESSFUL`，报告输出 `No issues found` |
| TC-BLD-004 | P0 | 执行 `:app:assembleRelease` 并验证签名 | 生成 signed APK/AAB，签名校验通过 |
| TC-BLD-005 | P0 | 安装后读取包信息 | 包名为 `com.remophoto`，versionCode/versionName 与 Release Notes 和 Tag 一致 |
| TC-BLD-006 | P1 | 检查发布提交内容 | 不包含密钥、`local.properties`、APK/AAB、本地日志、IDE/AI 缓存或私人测试数据 |

## 5. 通用真机证据

优先使用 `scripts/run-smoke.ps1` 生成统一证据。需要手工补充或定位单步失败时，可参考脚本输出目录中的 `smoke.log` 和对应 adb 命令记录。

每个 P0 真机用例至少记录：

- 日期、Commit、versionName/versionCode、APK SHA-256。
- 设备型号、Android 版本、安装方式。
- 数据集规模和网络条件。
- 实际结果、截图/录屏、Crash buffer。
- 关键 AppLogger 摘要；日志必须脱敏。

## 17. Alpha P0 Smoke 集合

个人开发者的每个 Alpha 候选包只要求完成以下最低集合，目标控制在一次构建加 15～30 分钟真机操作：

1. 自动门禁：`TC-BLD-002`～`TC-BLD-005`。
2. 安装与启动：`TC-APP-001`；有上一公开版本和现成数据时追加 `TC-APP-002`。
3. 本地核心闭环：`TC-LOC-001`、`TC-LOC-004`、`TC-LOC-006`。
4. 相册与图片：`TC-ALB-002`、`TC-ALB-005`、`TC-ALB-007`、`TC-GAL-001`、`TC-VIEW-001`、`TC-VIEW-002`。
5. 基本设置和诊断：`TC-SET-002`、`TC-OBS-002`，最后检查 Crash buffer。

其中 `scripts/run-smoke.ps1` 可自动覆盖 `TC-BLD-004` 的签名校验、`TC-BLD-005`、`TC-APP-001` 的安装启动基础证据、Crash buffer、UI dump、截图、启动日志、应用文件日志和基础日志隐私扫描。SAF 目录选择、扫描、本地相册浏览、全屏交互、备份和双机远程仍需按本节范围人工执行或通过后续专项自动化补齐；只有一台设备或本轮明确暂缓远程时，应把双机远程记录为未执行/暂缓，而不是失败。

首次公开 Alpha 还应各完成一次以下模块包：

## 19. 用例演进规则

- 功能扩展：向能力矩阵增加参数，并复用已有行为断言；只有行为不同才新增独立用例。
- 内部重构：只要用户行为和已发布契约不变，测试原则上不应因替换框架、存储或算法而修改。
- 有意变更：同一提交中更新功能规格、测试数据和期望；涉及持久化数据或公开协议时同时增加迁移/兼容用例。
- 功能废弃：用例先标记适用版本或 `RETIRED`，在兼容窗口结束后再移出活跃回归，避免直接删除历史依据。
- 缺陷回归：优先增加能复现根因的最小用例，不复制完整 E2E 流程。
- 常量治理：分页、TTL、大小上限和支持格式应有单一来源；测试通过参数读取或共享 fixture 使用，避免散落魔法数字。
- 契约演进：新增字段默认向后兼容，接收方应容忍未知可选字段；破坏性变化必须提升协议或格式版本。
- 多实现并存：使用仓库类型、协议版本或能力标签标记适用范围；同一行为共享用例，不要求所有实现具备完全相同的可选能力。

## 20. 发布退出标准
